# Prophet 모델 명세서 생성

보험약관(및 기초서류)을 읽어 FIS Prophet에서 그대로 따라 모델링할 수 있는 **모델 명세서 초안**을 만든다. 산출물은 *계산*이 아니라 *설계도*다 — 계산엔진(Prophet)에 들어가기 전, 변수·로직·약관근거를 구조화한 참고서.

## 핵심 원칙

- **초안만 만든다.** Prophet 문법 변환, 회사 경험가정 교체, 계리 검증, 선임계리사 사인오프는 사람이 한다. 절대 "완성된 모델"로 제시하지 않는다.
- **약관에서 확인된 사실만 쓴다.** 추측 금지. 약관에 없거나 불명확하면 "약관상 불명확"으로 표기한다.
- **가정값(요율·수익률·사업비)은 전부 샘플**로 두고 포뮬라에 `// 샘플 — 산출방법서·경험자료로 교체` 주석을 단다. 산출방법서 미보유로 근사한 항목은 의미란에 "근사"를 명시한다.
- **가정한 것은 전부 추적한다.** `// 샘플`·`(근사)`로 표시하는 데서 그치지 말고, **모든 가정을 한 곳에 모아 "가정 사항 명세(Assumptions Log)"** 로 별도 기록한다(아래 출력 형식 6번). 어떤 변수에서, 무엇을, 왜 가정했고, 무엇으로 교체해야 하는지 추적 가능해야 한다.
- **내용을 빠짐없이 전부 보여준다.** 영역·변수·가정을 요약·생략하지 않는다. 모든 영역의 변수표를 끝까지 싣고, 분량이 커도 잘라내지 않는다(필요하면 여러 메시지로 이어서 전부 출력).

## 워크플로 (3단계)

상세 규칙은 `references/conventions.md`를, 출력 형식·품질 기준은 `references/example-spec.md`를 먼저 읽고 따른다.

### 0. 입력 확인
사용자가 약관/기초서류 PDF를 올렸는지 본다. 없으면 "약관 PDF를 올려 주세요(없으면 상품 구조를 설명해 주셔도 됩니다)"라고 요청한다. 여러 파일이면 약관=담보 정의 근거, 사업방법서=상품 구조·갱신·면책 근거, 산출방법서=요율 근거로 구분해 읽는다.

### 1. 상품 핵심 사실 추출
약관·기초서류를 읽고 모델링에 필요한 사실을 **약관 조항 번호와 함께** 정리한다. 반드시 포함: 상품명·유형 / 담보 목록(주계약·특약, 의무부가·선택 구분, 각 지급사유) / 가입금액 기준 / 납입기간·보험기간·갱신 / 면책(대기)기간 / 최저보증(종류·기준) / 적립·연금 구조와 단계 / 사업비·해지환급금 조항. 이 요약은 **출력 맨 위에 "추출된 상품 사실(검증용)"** 으로 싣는다 — 사용자가 약관과 대조해 가장 먼저 검증하는 부분이다.

### 2. 모델링 영역 설계
1단계 사실을 바탕으로 모델링 영역(섹션)과 영역별 체크리스트 항목을 설계한다. 표준 골격을 상품 유형에 맞게 재구성한다(`conventions.md`의 영역 골격·상품유형별 적응 참조). 변액→특별계정·펀드 영역, 연금→연금개시 전·후 단계 분기, 암·건강→담보별 발생률 영역, 최저보증(GMxB)→보증 정의·비용·초과지급 영역.

### 3. 영역별 변수·포뮬라 생성
각 영역마다 변수를 설계한다. 각 변수 = `변수명` / `타입` / `의미`(약관 조항 근거 포함) / `포뮬라`(Prophet 유사 pseudo-code). 명명·타입·포뮬라 컨벤션은 `conventions.md`를 엄격히 따른다. 앞 영역에서 만든 변수명을 참조해 의존성을 일관되게 잇는다. K-IFRS17 영역은 BEL·RA·CSM과 CSM 롤포워드까지, 변액·연금은 VFA 기준으로 작성한다. 변수를 만들며 **샘플·근사 항목을 그때그때 기록**해 두었다가 4단계 가정 명세에 모은다.

### 4. 가정 사항 명세(Assumptions Log) 작성
3단계까지 `// 샘플`·`(근사)`로 표시한 항목과 문서 부재로 채우지 못한 항목을 **한 표로 모아** 분류한다. 세 분류로 나눈다:
- **ⓐ 샘플 가정값** — 산출방법서·경험자료로 교체해야 하는 요율·시나리오. 열: `# | 변수/항목 | 영역 | 가정한 값·내용 | 교체 근거(필요 자료)`
- **ⓑ 구조 근사** — 산출방법서 부재·OCR 손상으로 로직을 추정한 부분. 열: `# | 항목 | 영역 | 근사한 내용 | 확인 필요 원문`
- **ⓒ 문서·데이터 부재** — 아예 받지 못한 기초서류·통계. 열: `# | 부재 항목 | 영향 | 비고`

이 명세는 변수표의 샘플 주석과 **1:1로 대응**해야 한다(누락 금지). 누가 무엇을 검증·교체해야 하는지가 이 표만 보고 추적되도록 쓴다.

## 출력 형식

다음 순서의 문서로 출력한다. **기본은 화면에 전문을 그대로 다 보여주고, 사용자가 원하거나 분량이 크면 .md/.docx 파일로도 저장**한다:

1. 제목 + 상품명 + 태그 + (영역 수·변수 수)
2. **추출된 상품 사실 (약관 검증용)** — 약관 조항 포함
3. 영역별 섹션. 각 섹션은 `변수명 | 타입 | 의미 | 포뮬라` 표. **모든 영역을 빠짐없이.**
4. 포뮬라 컨벤션 각주
5. **가정 사항 명세 (Assumptions Log)** — ⓐ샘플 / ⓑ근사 / ⓒ부재 3분류 표. (4단계 산출물)
6. 경고: 가정값은 샘플, AI 초안이며 선임계리사 검증·거버넌스를 대체하지 않음.

분량이 크면 영역을 나눠 차례로 생성하되, 한 영역도 빠뜨리지 않는다. 요약·생략으로 내용을 줄이지 않는다.

## docx / md 파일 저장

- **md**: 위 출력 형식 그대로 마크다운 표로 저장한다.
- **docx**: `scripts/docx_builder.py`(데이터 주도형 빌더)를 사용한다 — 직접 docx 서식 코드를 새로 짜지 말 것.
  1. 위 6개 구성요소를 빌더 스키마에 맞춘 **JSON**(`spec.json`)으로 만든다. 스키마·키는 `scripts/docx_builder.py` 상단 docstring 참조(`facts`/`areas`/`assumptions.sample|approx|missing`/`conventions`/`warnings`).
  2. 실행: `python <스킬경로>/scripts/docx_builder.py spec.json 출력.docx`
  3. 빌더는 값에 `// 샘플`·`샘플`·`근사`가 있으면 자동으로 빨강 강조하고, 가정 명세의 변수명 열도 강조한다. 한글 폰트(맑은 고딕)·미래에셋 색상·표지·페이지 구분을 자동 적용한다.
  - 환경에 `python-docx`가 없으면 `pip install python-docx` 후 사용. pandoc은 필요 없다.

## 하지 말 것

- 실제 BEL/CSM 등 **숫자를 산출하지 않는다** (틀만 만든다).
- 약관에 없는 담보·요율을 지어내지 않는다.
- 산출물을 검증·승인된 모델로 표현하지 않는다.
